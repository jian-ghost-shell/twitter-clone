# Gemini Suggestions for Twitter Clone

> 以下是针对该 Twitter Clone 项目的综合改进建议汇总，分为架构解耦、性能体验以及数据安全三个维度。

---

## 一、架构解耦与规范化（实现高内聚、低耦合）

- [ ] **引入后端 Service 逻辑层**
  - 将核心业务逻辑（如复杂的推文查询、交互状态计算）从 Next.js HTTP 路由中剥离，使其独立于请求对象，提升可测试性与跨端复用性。

- [ ] **封装前端 Custom Hooks 表现层**
  - 将数据流状态（State）、网络请求和副作用（如无限滚动监听）抽离为自定义 Hooks（如 `useTweets`），让 UI 组件纯粹化，解决"巨型组件"问题。

- [ ] **抽象前端 API 客户端层**
  - 建立统一的网络请求服务，避免在 UI 组件内部散落大量硬编码的原生 `fetch` 路径，降低 UI 与特定 API 结构的耦合。

- [ ] **消除冗余逻辑**
  - 全面盘点并使用已抽象的工具（例如强制在组件中使用已封装好的 `useRealtime` Hook，而不是在组件内重新手写一遍 WebSocket 订阅流程）。

---

## 二、性能与体验进阶优化

- [ ] **升级状态请求管理**
  - 引入 React Query 或 SWR 替代原生的 `fetch + useState`，一揽子解决请求去重、数据缓存和状态同步问题。

- [ ] **实现长列表虚拟化 (Virtualization)**
  - 配合无限滚动，引入虚拟滚动库（如 `@tanstack/react-virtual`），仅渲染视口内的 DOM 节点，防止刷推文过多导致低端设备卡顿。

- [ ] **视觉加载反馈升级**
  - 将简单的文本加载提示（"Loading more..."）替换为 Twitter 风格的骨架屏（Skeleton）或占位动画，提升产品的细腻度。

---

## 三、数据存储与安全策略优化

- [ ] **迁移图片存储方案**
  - 废弃直接在数据库存储 Base64 图片数据的做法，改用外部云存储（如 AWS S3、Vercel Blob 或 Cloudinary），数据库仅保留 URL，避免性能瓶颈。

- [ ] **增强身份验证防护**
  - 目前的"免感知自动注册"体验虽好，但需引入防机器制（如验证码或简单的频控限制），防止接口被滥用产生大量恶意僵尸账号。
